# AI Coding 效能度量方案 v4.9 — 总体思路

> 版本: v4.9 | 2026年4月
> 核心原则: 最小改动工作流 + 硬指标有数据支撑 + 考核到团队、个人仅观测

---

## 一、顶层设计

### 1.1 要解决什么问题

企业推广 AI 编码工具后，面临三大管理盲区：

1. **用了多少** — 不知道 AI 在团队中的实际渗透率
2. **用得怎样** — AI 是否真正提升了效能，还是流于形式
3. **怎么考核** — 不同岗位 AI 使用场景差异大，如何公平量化

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **最小改动** | 开发者日常操作不增加负担，Hook 自动 + 任务关闭时一次标注 |
| **客观可验** | 代码级数据来自 git diff 行统计，非人工估算 |
| **双轨互补** | 任务级覆盖全员，代码级精准归因到编码人 |
| **可下钻** | 支持研发中心 → 部门 → 项目团队 → 个人四级穿透 |

## 二、核心机制：双轨标注

### 2.1 第一轨 — 任务级标注（云效 Projects）

```
打标单元：任务（Task）
打标人：任务执行人
打标时机：
  第一步：任务开始时 → 填写意图（预计 AI 参与程度）
  第二步：任务关闭时 → 确认或修正（以实际为准）
```

**两个字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| AI 参与度 | 枚举四档 | [AI主导]≥70%：核心逻辑由 AI 生成，人工审查修改 / [AI协同]30-70%：AI 与开发者共同完成，各有主要贡献 / [AI辅助]10-30%：开发者主导，AI 补充修改少量内容 / 无：未使用 AI |
| 工作场景 | 枚举十类 | 编码开发、需求分析、架构升级、代码优化、测试、运维、逆向安全、算法模型、文档知识、其他 |

**价值**：不仅知道"用了多少 AI"，还知道"在什么场景用了 AI"。

### 2.2 第二轨 — 代码级标注（Git Hook 自动）

```
标记方式：AI 工具生成代码时自动插入
  // AI-GENERATED-BEGIN (by Tongyi Lingma)
  ... AI 生成的代码 ...
  // AI-GENERATED-END

统计方式：prepare-commit-msg Hook 在 commit 时自动触发
  1. 扫暂存区 diff 中的 BEGIN/END 标记
  2. 计算 AI 标记行数占比
  3. 自动注入 commit 消息前缀：[AI协同](ai:42/total:98/ratio:42.9%)

归因：commit author = 实际写代码人，不受 MR 操作人影响
```

**三工具支持：**

| 工具 | 规则文件 | 说明 |
|------|---------|------|
| Claude CLI | `CLAUDE.md` | 完全支持 |
| Trae | `.trae/rules/ai-mark.md` | v1.3.0+ 支持 |
| 通义灵码 | `.lingma/rules/ai-mark.md` | 仅对话/AI程序员模式 |

### 2.3 两轨关系

```
任务级（主观 + 覆盖广）              代码级（客观 + 精准）
    ├── 覆盖全员                          ├── 仅编码岗位
    ├── 人工填写（开始意图 + 关闭确认）      ├── Hook 自动
    ├── 含非编码场景                       ├── 精确行数统计
    └── 以关闭日期归月                     └── 以提交日期归月
         │                                      │
         ▼                                      ▼
  支撑任务层面指标                        支撑代码层面指标
  ├── 任务 AI 参与率                      └── AI 代码生成占比
  ├── 综合提效（任务产出效率）
  └── 全员 AI 使用率（OR 逻辑）←──────────────┘
                                    （编码岗兜底）
```

> 两轨各司其职，不是对同一指标的双重验证：任务级覆盖全员全场景，代码级精确归因到行；全员 AI 使用率是唯一同时依赖两轨的指标（双轨 OR 判定）。

## 三、指标体系

### 3.0 目标设定背景：软件开发人员时间分布

指标目标值的设定基于行业经验数据，确保目标有据可依：

| 工作类型 | 时间占比 | AI 渗透难度 |
|---------|:-------:|:----------:|
| 编码开发（含单测、代码审查） | 30-40% | 低（工具成熟） |
| 需求理解、方案设计、技术讨论 | 20-25% | 中 |
| 调试、排障、联调 | 15-20% | 中 |
| 会议、沟通协作 | 10-15% | 高 |
| 文档、知识整理 | 5-10% | 低 |
| 其他（部署、运维等） | 5-10% | 中 |

**对目标设定的影响：**
- 编码类任务占全部任务约 30-40%，AI 渗透最容易，H1 任务 AI 参与率 ≥30% 即要求编码类任务基本覆盖
- H2 任务 AI 参与率 ≥60%，意味着编码类全覆盖的同时，需求/设计/文档等场景也开始渗透，推动 AI 从"编码工具"向"全场景助手"延伸
- AI 代码生成占比直接对应编码类工作，H1 ≥30% / H2 ≥40% 与编码工作占比匹配

### 3.1 核心考核指标

| 指标 | 公式 | H1目标 | H2目标 | H3目标| 数据源 |
|------|------|:------:|:------:|--------|
| **AI 代码生成占比** | AI标记行 / 总变更行 | ≥30% | ≥40% | ≥50% |Git Hook 自动 |


**关键设计**：四档标签**不参与 AI 代码生成占比的计算**（占比直接按 Git 行数统计，客观可验证）；但四档标签**参与任务 AI 参与率的统计**（非"无"任务数 / 总任务数），用于衡量 AI 在任务层面的覆盖广度。

> **综合提效仅考核到项目团队**，不考核到人（工时个人间不可比），不考核到部门（平均数掩盖差异）。部门/研发中心可看统计值，用于趋势观察、资源调配。

**为什么不考核到人：**
- 工时填报粒度因人而异，个人间不可比
- 任务分配不均（TL 分配差异），个人产出率失真
- 引导刷量风险（拆小任务、挑简单任务）

**为什么不考核到部门：**
- 部门是管理汇总层，没有独立业务目标
- 平均数掩盖团队差异（A 团队 60% + B 团队 20% = 平均 40%）
- 责任无法落实（部门领导不写代码、不关任务）

### 3.2 考核辅助指标

| 指标 | 说明 | H1目标 | H2目标 | 考核层级 |
|------|------|:------:|:------:|:--------:|
| 任务 AI 参与率 | 非"无"任务数 / 总任务数 | ≥30% | ≥60% | 项目团队（考核） |
| AI 评审开启率 | 已开启 AI 代码评审的仓库数 / 团队总仓库数 | ≥80% | =100% | 项目团队（考核，仅 Codeup；未使用 Codeup 的团队由部门评估是否纳入考核） |
| 全员 AI 使用率 | AI 活跃人数 / 总人数（活跃定义：当月有 AI 标记 commit ≥1 **或** 任务 AI 参与度非"无" ≥1） | ≥80% | =100% | 项目团队（考核） |

### 3.3 观察指标（不考核）

两个质量指标互补，共同反映 AI 使用对代码质量的影响：

| 指标 | 计算方式 | 侧重 | 用途 |
|------|---------|------|------|
| 缺陷重开率 | 重开缺陷数 / 总关闭缺陷数 | 修复质量 | 反映"修了没修好"，不受任务粒度影响，跨团队可比 |
| 缺陷密度 | 当月新增缺陷数 / 当月关闭任务数 | 开发质量 | 反映"做了多少任务产生了多少缺陷"，仅纵向趋势对比，不跨团队横比 |

> 缺陷密度受任务拆分粒度影响，不同团队间不可横向对比；但同一团队纵向趋势有参考价值，可观察 AI 使用量提升后缺陷密度是否同步变化。

## 四、考核体系

### 4.1 考核层级总览

所有硬指标**仅考核到项目团队**，个人数据**仅观测，不与绩效挂钩**。

| 层级 | AI代码占比 | 任务AI参与率 | 综合提效 | 全员使用率 | AI评审开启率 | 质量观察① | 说明 |
|------|:---------:|:-----------:|:-------:|:---------:|:-----------:|:---------:|------|
| 个人 | 观测 | 观测 | — | — | — | 观测 | TL参考，不考核 |
| 项目团队 | ✅ 考核 | ✅ 考核 | ✅ 考核 | ✅ 考核 | ✅ 考核（仅Codeup） | 观察 | 业务闭环最小单元 |
| 部门 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 不考核 |
| 研发中心 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 展示统计值 | 不考核 |

> ① 质量观察包含两个指标：缺陷重开率（跨团队可比）+ 缺陷密度（仅纵向趋势）。

### 4.2 个人观测数据

个人观测数据统一展示，**不分岗位、不设权重、不设评级**：

| 观测指标 | 公式 | 说明 |
|----------|------|------|
| 个人 AI 代码占比 | 个人 AI 标记行 / 个人总变更行 | 客观行数统计 |
| 个人任务 AI 参与率 | 个人负责的非"无"任务数 / 个人总任务数 | 任务标注统计 |
| 个人 AI 评审覆盖率 | 个人 MR 中有 AI 评审的占比 | 仅 Codeup 团队 |
| 个人场景分布 | 按「工作场景」字段分组统计 | 反映 AI 使用广度 |

> 个人数据供 TL 参考，用于发现落后成员、针对性辅导。
> 不与个人绩效直接挂钩，减少推广期阻力。

### 4.3 团队评级

```
团队指标（项目团队级）：
                        H1目标      H2目标     H3 目标
  ├── AI 代码占比        ≥ 30%       ≥ 40%      ≥ 50%
  ├── 综合提效           ≥ 9%        ≥ 12%      ≥ 15%
  ├── 任务 AI 参与率     ≥ 30%       ≥ 45%      ≥ 60%
  ├── 全员使用率         ≥ 80%       = 90%      = 100%
  └── AI 评审开启率      ≥ 80%       = 90%      = 100%（仅 Codeup）

质量观察（不考核）：
  ├── 缺陷重开率（跨团队可比，仅观察，不设目标）
  └── 缺陷密度 = 新增缺陷数/关闭任务数（仅纵向趋势，不跨团队横比）

评级：★★★★★ 优秀 / ★★★★ 良好 / ★★★ 达标 / ★★ 待改进

组织层级说明：
  项目团队 → ★ 考核主体，有达标线和评级
  部门     → 仅展示统计值，不考核
  研发中心 → 仅展示统计值，不考核
```

### 4.4 个人观测卡示例

```
张三 · 后端开发 · 2026年X月（观测，不考核）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  AI 代码占比: 46.4%  (团队均值 40%)
  任务 AI 参与率: 80%  (团队均值 65%)
  AI 评审覆盖率: 75%  (团队均值 70%)
```

> 个人数据卡标注"观测"字样，不与绩效挂钩。

### 4.5 月度归属规则

| 数据 | 归属依据 | 原因 |
|------|---------|------|
| 任务 AI 参与率 | 任务**关闭日期** | 以实际完成为准 |
| AI 代码占比 | commit **提交日期** | 跟随编码行为 |
| 任务产出效率 | 任务**关闭日期** | 跨月任务计入关闭月 |
| 缺陷重开率 | 任务**关闭日期** | 以实际完成为准 |
| 在途任务 | 不计入当月 | 避免未完成任务干扰考核 |

## 五、质量保障

```
质量不靠交叉校验，每个指标独立可信：

TL 月度抽检 → 每月 5 个高 AI% commit，人工核实标记与代码是否匹配
产出异常检测 → AI% 突增但产出下降 → 自动告警
趋势横向对比 → 个人 AI% 显著偏离团队均值 → 关注
AI 评审覆盖率 → AI 生成的代码是否经过 AI 评审
缺陷重开率 → 代码质量是否因 AI 使用而下降（综合提效质量维度）
```

## 六、数据采集架构

### 6.1 两类团队

| 团队类型 | 任务级 | 代码级 | 采集方式 |
|---------|--------|--------|---------|
| 云效 Codeup | 自动 | 自动 | API 全自动采集 |
| 公司 Git | 自动 | Hook自动统计 + 手工上报 | 任务自动 + 月度手工填报 |

> 公司 Git 团队同样安装 Git Hook，Hook 在本地自动统计 AI 标记行数；但因无 API 对接，统计结果由 TL 每月手工汇总填报至度量平台。

### 6.2 数据源统一

| 数据项 | 来源 | 采集方式 |
|--------|------|---------|
| 任务 AI 参与度 | Projects | API 自动 |
| 任务关闭数 | Projects | API 自动 |
| 投入工时 | Projects | API 自动 |
| 缺陷重开率 | Projects | API 自动 |
| AI 代码占比 | Git Hook + git log | Hook 自动 |
| AI 评审覆盖率 | Codeup MR API | API 自动（仅 Codeup） |

> 综合提效（任务产出效率）来自云效 Projects，一套 API 搞定。

### 6.3 手工填报（公司 Git 团队）

每月最后一日前由项目 TL 填写，共四张表：
1. 项目基本信息
2. Commit 级 AI 数据（按人：AI 标记行/总变更行）
3. 提效数据（关闭任务数/投入工时/缺陷重开率 — 全部 API 自动取）
4. AI 使用情况明细（可选，用于案例沉淀）

> 手工表格式与 API 采集格式对齐，确保计算公式一致、团队间可横向对比。

## 七、基线建立

启动前取 **1月、2月、3月三个自然月的月平均值** 作为基线：

```
每月产出效率 = 当月关闭任务数 / 当月投入工时
基线任务产出效率 = 3个月产出效率的平均值
基线缺陷重开率   = 3个月重开率的平均值
```

| 基线项 | 取数方式 | 层级 |
|--------|---------|:--------:|
| 任务产出效率（任务数/工时） | Projects API 按月统计后取平均 | 项目团队（考核） |
| 缺陷重开率 | Projects API 按月统计后取平均 | 项目团队（仅观察） |

**考核时计算：**

```
综合提效 = 本月任务产出效率 / 基线任务产出效率 - 1
```

| 特殊情况 | 处理方式 |
|---------|---------|
| 新项目（无历史数据） | 以试跑阶段（P1 第2-3周）数据作为基线 |
| 历史数据不足3个月 | 有几个月算几个月，至少1个月 |

基线确定后本年度不调整。

## 八、实施路径

```
P0 配置（第1周）     Projects 字段 + Git Hook + 工具规则 + 基线建立
  ↓
P1 试跑（第2-3周）   后端团队先行，适应期不考核，TL 带头
  ↓
P2 全覆盖（第4-6周） 全员接入，开始正式统计
  ↓
H1 评估（6.30）      中期考核，AI 代码占比≥30%
  ↓
H2 总评（12.30）     年终考核，AI 代码占比≥40%
```

## 十、一句话总结

> 通过**任务级人工标注**（覆盖全员、全场景）与**代码级自动标记**（精确行数、零操作）的双轨机制，以**AI 代码行占比**和**综合提效**两大硬指标为核心，分岗位、分团队、可下钻、可考核，实现 AI 编码效能的**最小干预、最大客观**的量化管理。
